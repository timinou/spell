#+TITLE: Spell Org Query Language Design
#+CREATED: [2026-03-14]
#+AUTHOR: Spell Engineering

* Overview

Spell's query layer has two execution paths. The choice is transparent to the
caller — routing is automatic based on query complexity.

- *TS path*: structural queries over parsed org items. No IPC overhead.
  Handles state, tags, property equality, priority, text search.

- *Emacs/org-ql path*: full org-ql expressiveness, arbitrary Lisp predicates.
  Required for date ranges, clocked time, effort filters, and numeric property
  comparisons. The Emacs bridge is always available; these queries run via the
  MCP JSON-RPC socket.

Routing rule: =requiresEmacs(filter)= returns true → Emacs path. Otherwise
TS path. No caller configuration needed.

* org-ql Query Language

** What org-ql Is

org-ql is a Lisp-based query DSL for org files. Queries are S-expressions
evaluated against org item nodes. Results are lists of org elements (full AST
nodes), not strings.

#+BEGIN_EXAMPLE
;; All high-confidence facts that haven't decayed
(org-ql-select "memory.org"
  '(and (todo "FACT")
        (property "CONFIDENCE" :predicate
          (lambda (v) (> (string-to-number v) 0.7)))
        (not (ts-inactive :to today)))
  :action #'element)
#+END_EXAMPLE

** Core org-ql Predicates

| Predicate              | Description                                    |
|------------------------+------------------------------------------------|
| =(todo "STATE")=       | Match items with given TODO state              |
| =(tags "tag1" "tag2")= | Match items carrying all listed tags           |
| =(property "K" "V")=   | Match items where property K equals V          |
| =(heading "text")=     | Match items where headline contains text       |
| =(ts :from D :to D)=   | Active timestamp in range                      |
| =(ts-inactive ...)=    | Inactive timestamp in range                    |
| =(deadline ...)=       | DEADLINE timestamp predicate                   |
| =(scheduled ...)=      | SCHEDULED timestamp predicate                  |
| =(effort >= "2h")=     | Effort property comparison                     |
| =(clocked :on D)=      | Clocked time on a specific date                |
| =(level N)=            | Match items at outline depth N                 |
| =(children ...)=       | Match items with children matching predicate   |
| =(ancestors ...)=      | Match items with ancestors matching predicate  |
| =(not ...)=            | Negate a predicate                             |
| =(and ...)=            | All predicates must match                      |
| =(or ...)=             | Any predicate must match                       |

** Custom Predicates

Custom org-ql predicates registered via =org-ql-defpred=.

*** confidence-above

#+BEGIN_EXAMPLE
(org-ql-defpred confidence-above (threshold)
  "Match items with CONFIDENCE property above THRESHOLD."
  :body
  (when-let ((val (org-entry-get nil "CONFIDENCE")))
    (> (string-to-number val) threshold)))
#+END_EXAMPLE

Usage: =(confidence-above 0.7)=

*** decays-after

#+BEGIN_EXAMPLE
(org-ql-defpred decays-after (date)
  "Match items that have not yet decayed as of DATE."
  :body
  (when-let ((decay (org-entry-get nil "DECAYS_AT")))
    (org-time>= decay date)))
#+END_EXAMPLE

Usage: =(decays-after "today")=

*** session-agent

#+BEGIN_EXAMPLE
(org-ql-defpred session-agent (agent-pattern)
  "Match session items with AGENT property matching AGENT-PATTERN."
  :body
  (when-let ((agent (org-entry-get nil "AGENT")))
    (string-match-p agent-pattern agent)))
#+END_EXAMPLE

Usage: =(session-agent "code-.*")=

* TypeScript Query Builder

** OrgQlFilter Interface

#+BEGIN_SRC typescript
export interface OrgQlFilter {
  todo?: string[];
  tags?: string[];
  priority?: { op: ">=" | "<=" | "="; value: string };
  properties?: Array<{ key: string; value: string; op?: "=" | ">" | "<" }>;
  dateRange?: { from?: string; to?: string };
  clocked?: { on?: string; from?: string };
  effort?: { op: "<=" | ">=" | "="; value: string };
  text?: string;
  and?: OrgQlFilter[];
  or?: OrgQlFilter[];
  not?: OrgQlFilter;
}
#+END_SRC

** requiresEmacs

#+BEGIN_SRC typescript
// Returns true when the filter needs org-ql via Emacs.
// Simple fields (todo, tags, text, priority, property equality) are TS-side.
// dateRange, clocked, effort, and numeric property ops require Emacs.
export function requiresEmacs(filter: OrgQlFilter): boolean {
  if (filter.dateRange || filter.clocked || filter.effort) return true;
  if (filter.properties?.some(p => p.op && p.op !== "=")) return true;
  if (filter.and?.some(requiresEmacs)) return true;
  if (filter.or?.some(requiresEmacs)) return true;
  if (filter.not && requiresEmacs(filter.not)) return true;
  return false;
}
#+END_SRC

** Keyword Syntax

The =query= parameter accepts keyword syntax for common filters:

| Keyword             | Example               | Maps to                     |
|---------------------+-----------------------+-----------------------------|
| =todo:STATE=        | =todo:DOING=          | filter.state                |
| =tags:t1,t2=        | =tags:auth,backend=   | org-ql =(tags ...)= via TS  |
| =priority:>=B=      | =priority:>=B=        | filter.priority             |
| =property:K=V=      | =property:LAYER=api=  | filter.properties           |

Advanced predicates (=effort:>=2h=) are not yet in keyword syntax. Use the
=ql= parameter for those.

** Raw org-ql Passthrough

The optional =ql= parameter accepts a raw org-ql sexp string. It bypasses
keyword parsing entirely and goes straight to the Emacs org-ql-query tool.
Intended for internal subsystems (memory consolidation, planning) that need
precise org-ql queries without going through keyword syntax.

#+BEGIN_SRC typescript
// Example: find items clocked today with effort >= 1h
tool.execute({
  command: "query",
  ql: "(and (clocked :on today) (effort >= \"1h\"))",
});
#+END_SRC

* Routing

#+BEGIN_SRC
query(filter)
  │
  ├── filter.ql present?
  │     └── YES → org-ql via Emacs (raw sexp passthrough)
  │
  ├── filter.query present?
  │     └── parseKeywordQuery → OrgQlFilter
  │           └── requiresEmacs?
  │                 ├── YES → buildOrgQlSexp → org-ql via Emacs
  │                 └── NO  → applyFilter (TS)
  │
  └── structural filter only → applyFilter (TS)
#+END_SRC

* Emacs Integration Protocol

** Transport

org-ql queries are executed via the Emacs MCP JSON-RPC socket (Unix domain
socket, socat bridge). Each call opens a fresh connection. The tool name is
=org-ql-query=; arguments are ={ files: string[], query: string }=.

** Error Handling

Emacs subprocess failures must surface with full fidelity. A failed org-ql
query must not return an empty result set — it must throw. Empty set and
failure are not the same thing.

Failure modes:
- socat not on PATH → throw from =createOrgClient=
- org-ql expression rejected by Emacs → throw from JSON-RPC error response
- Socket connection refused → throw from Bun.spawn
- Emacs timeout (> 30s) → throw with timeout message

None of these should silently degrade to an empty result.

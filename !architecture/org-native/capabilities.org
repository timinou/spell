* What org-native means

Spell's knowledge — memories, plans, tasks, skills, sessions — lives in
org-mode files.  Not as a storage format.  As a *queryable knowledge graph*
with typed properties, timestamps, tags, clocking, and a query language that
works without Emacs but scales to full org-ql when Emacs is present.

Every piece of project knowledge the agent creates is a first-class org item:
searchable, filterable, cross-referenced, and editable by both the agent and
the user in any org-aware editor.

* Queryable memory with confidence

Memories are org headings with a ~CONFIDENCE~ property (0.0–1.0).  The agent
can ask "what do I know about authentication with confidence above 0.7?" and
get a filtered, ranked answer — not a blob of markdown to re-read.

#+begin_example
\* Auth uses JWT with RS256 signing               :auth:security:
:PROPERTIES:
:CONFIDENCE:     0.92
:SCOPE:          project
:SOURCE_SESSION: SESS-014-refactor-auth
:LAST_VALIDATED: [2026-03-12]
:END:

RSA key pair loaded from AUTH_PRIVATE_KEY env var.
Tokens expire after 1h; refresh tokens after 30d.
#+end_example

What this enables:
- *Selective recall*: query memories by tag, scope, confidence threshold, or
  recency — instead of injecting the entire memory file into context.
- *Confidence decay*: memories not reinforced across sessions can fade,
  preventing stale knowledge from polluting decisions.
- *Session provenance*: every memory traces back to the session that created
  it via ~SOURCE_SESSION~, so the agent (or user) can audit where a belief
  came from.
- *User curation*: open ~MEMORY.org~ in Emacs, adjust confidence scores,
  retag entries, add notes.  The agent respects your edits.

* Structured query language

A keyword syntax works everywhere — no Emacs, no special setup:

#+begin_example
todo:DOING tags:auth priority:>=B
confidence:>0.7 category:tasks
#+end_example

| Query                  | What it finds                                    |
|------------------------+--------------------------------------------------|
| ~todo:DOING~           | All in-progress items across all categories      |
| ~tags:auth+security~   | Items tagged with both ~auth~ and ~security~     |
| ~priority:>=B~         | High-priority items (A or B)                     |
| ~confidence:>0.7~      | High-confidence memories                         |
| ~category:tasks~       | Only items from the tasks category                |

Omit category to search *everything* — plans, tasks, memories, sessions — in
one query.  The agent uses this to answer questions like "what's blocking the
auth refactor?" by pulling context from multiple knowledge domains at once.

When Emacs is available, queries compile to org-ql s-expressions and gain
access to date math, deadline warnings, inherited tags, and custom predicates.
When it's not, the same query runs in-process with TypeScript.  The routing is
automatic and transparent.

* Cross-entity knowledge graph

Items reference each other through properties and tags:

- A *plan* carries a ~SESSION~ property linking to the session that created it.
- A *session* carries ~CLOCK~ entries tracking wall-clock time spent.
- A *memory* carries ~SOURCE_SESSION~ tracing its origin.
- A *task* carries ~DEPENDS~ and ~BLOCKS~ linking to other tasks.

This means:
- "Show me everything related to today's session" — one query.
- "What plans did I create this week, and how long did each session take?" —
  cross-reference plans with session clocking data.
- "Which memories came from sessions that also produced tasks?" — graph
  traversal across entity types.

The knowledge isn't siloed.  Plans know about sessions.  Sessions know about
tasks.  Memories know where they came from.

* Skills in org

Skills can be authored as ~SKILL.org~ files with org-native metadata:

#+begin_example
#+name: deployment
#+description: Kubernetes deployment patterns for this project
#+globs: k8s/** helm/**

\* Deployment skill content here...
#+end_example

What this enables:
- Skill metadata lives in ~#+KEY: value~ keywords — the same format the rest
  of the org system uses.  No YAML frontmatter, no format mismatch.
- Skill content can include org features: TODO items, PROPERTIES drawers,
  tables, source blocks with tangle targets.
- Skills are discoverable via the same query layer as everything else: "show
  me all skills tagged ~:k8s:~" works.

Markdown skills (~SKILL.md~) remain fully supported.  When both exist in the
same directory, the org version takes precedence.

* Session tracking

Every agent session can be an org item:

#+begin_example
* DONE SESS-042-auth-refactor                    :auth:refactor:
:PROPERTIES:
:CUSTOM_ID: SESS-042-auth-refactor
:MODEL:     claude-sonnet-4-20250514
:TOKENS_IN: 145230
:TOKENS_OUT: 23410
:END:
:LOGBOOK:
CLOCK: [2026-03-14 Fri 14:02]--[2026-03-14 Fri 14:47] =>  0:45
:END:
#+end_example

What this enables:
- *Effort tracking*: org's built-in ~CLOCK~ entries give wall-clock time per
  session.  Aggregate with ~org-clock-report~ or query programmatically.
- *Cost attribution*: token counts as properties let you answer "how much did
  the auth refactor cost across all sessions?"
- *Session history*: query past sessions by date, tag, model, or state.
  "Show me all sessions from this week that touched auth" — one query.
- *Plan linkage*: plans created during a session link back to it.  You can
  trace any plan to the session context where it was conceived.

* Org-ql power queries (with Emacs)

When Emacs is present, the full org-ql query engine is available with custom
predicates:

#+begin_example
;; Memories fading below threshold, not validated in 30 days
(and (confidence-above 0.3)
     (not (confidence-above 0.7))
     (ts-inactive :from "-30d"))

;; Overdue tasks with high priority
(and (todo "TODO" "DOING")
     (priority >= "B")
     (deadline :to today))

;; Everything from a specific session
(property "SOURCE_SESSION" "SESS-042-auth-refactor")
#+end_example

The ~confidence-above~ predicate is custom — it reads the ~CONFIDENCE~
property and compares numerically.  More custom predicates can be added to
~org-ql-query.el~ as the knowledge model grows.

Without Emacs, the TS query path handles the common cases (state, tags,
priority, property value matching).  The system tells you when a query needs
Emacs rather than silently returning degraded results.

* What the agent sees vs. what you see

The agent reads org files as raw org syntax through internal URL protocols
(~org://~, ~memory://~, ~skill://~, ~local://~).  It understands org natively
— no conversion to markdown happens in the agent-facing path.

You see the same files in your editor.  Emacs gives you folding, agenda views,
clock reports, and refiling.  Any other editor gives you plain text that's
still readable and grep-able.

The contract: the agent writes valid org.  You can edit what it wrote.  The
next query picks up your edits.  No sync step, no import/export, no format
conversion.  One source of truth in a format both humans and the agent can
work with directly.

* Design Rationale

** Why org, not a database

Org files are plain text, diffable, version-controlled, and human-editable.
A SQLite database would be faster to query but opaque to inspect, impossible
to hand-edit, and a merge conflict nightmare in version control.

The agent's knowledge should be as transparent as its code.  Org gives that
while still supporting structured queries via org-ql.

** Why dual-path queries (TS + Emacs)

Not every machine has Emacs.  The TS path ensures the agent works on CI,
containers, and remote servers with only the Bun runtime.  Emacs is the
high-fidelity path for users who have it — date math, inherited tags,
custom predicates, agenda integration.

The system is explicit about which path it takes.  ~requiresEmacs()~ returns
true when a query needs Emacs features.  Callers never silently get degraded
results.

** Why raw org to the LLM, not markdown conversion

1. LLMs understand org syntax natively.
2. Conversion is lossy — PROPERTIES, CLOCK, tags have no markdown equivalent.
3. Round-tripping (read + reason + edit) without format drift requires the
   agent to work in the source format, not a derived one.

** Why MEMORY.org alongside MEMORY.md (for now)

MEMORY.md is the existing contract.  MEMORY.org is additive — it carries
structured metadata that markdown can't express.  Once the consolidation
prompt reliably emits ~memory_entries~, MEMORY.md becomes derivable from
MEMORY.org and can be generated on demand rather than stored.

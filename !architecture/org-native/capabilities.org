* Overview

This document describes the org-native capabilities shipped in the Phase 0–6
migration.  It is intended for both contributors extending the org package and
for the agent runtime, which reads ~!architecture/~ as part of its context.

Each section names the capability, its location in the codebase, the public
API surface, and any concrete next steps that remain.

*** Example

#+begin_src typescript

// kw.title === "Foo", kw.globs === "src/**"

** DONE Keyword Query Syntax + Query Builder
:PROPERTIES:
:EFFORT:   3h
:FILES:    packages/org/src/query-builder.ts
:END:

A lightweight query language parsed entirely in TypeScript.  No Emacs required
for common predicates.  When Emacs is available, the filter compiles to an
org-ql s-expression.

*** Keyword syntax

#+begin_example
todo:DOING tags:auth priority:>=B confidence:>0.7
#+end_example

Supported tokens:

| Token              | Meaning                                |
|--------------------+----------------------------------------|
| ~todo:STATE~       | Match ~TODO~ keyword                   |
| ~tags:t1+t2~       | All listed tags present (~+~ separator)|
| ~priority:>=B~     | Priority comparison (A/B/C)            |
| ~confidence:>N~    | CONFIDENCE property (memory entries)   |
| ~category:NAME~    | Category prefix filter                 |

*** API

#+begin_src typescript
// packages/org/src/query-builder.ts

export interface OrgQlFilter {
  states?: string[];
  tags?: string[];
  priority?: { op: ">=" | "<=" | "=" | ">" | "<"; value: string };
  confidence?: { op: ">=" | "<=" | "=" | ">" | "<"; value: number };
  category?: string;
  raw?: string;  // pass-through sexp for Emacs-only predicates
}

/** Compile filter to an org-ql s-expression string. */
export function buildOrgQlSexp(filter: OrgQlFilter): string;

/** True when the filter requires Emacs (contains raw or unsupported predicates). */
export function requiresEmacs(filter: OrgQlFilter): boolean;

/** Parse "todo:DOING tags:auth priority:>=B" → OrgQlFilter. */
export function parseKeywordQuery(input: string): OrgQlFilter;
#+end_src

*** Routing contract

~requiresEmacs(filter) === false~ → ~applyFilter~ in ~org-reader.ts~ handles
it in-process.

~requiresEmacs(filter) === true~ → dispatch to ~org-ql-query~ MCP tool (Emacs
bridge).  Falls back to in-process with degraded fidelity if Emacs is
unavailable.

(org-ql-defpred confidence-above (threshold)
  "Match entries whose CONFIDENCE property exceeds THRESHOLD."
  :body (when-let ((v (org-entry-get nil "CONFIDENCE")))

* Design Rationale and Tradeoffs

** Why dual MEMORY.md + MEMORY.org

The markdown file is the established contract: prompts, consolidation output,
and the memory protocol all assume it exists.  Replacing it outright would
break every running session that hasn't upgraded.  The org file is additive —
it carries structured metadata (confidence, scope, session linkage) that
markdown cannot express without fragile conventions like YAML frontmatter
inside fenced blocks.

The cost is two files representing overlapping content.  The exit condition is:
once the consolidation prompt reliably emits ~memory_entries~, ~MEMORY.md~
becomes derivable from ~MEMORY.org~ (render org to markdown) and can be
generated on demand rather than stored.

** Why vendor org-ql instead of depending on user's Emacs packages

Spell targets machines where Emacs is installed but not necessarily configured
as a package-managed environment.  Requiring ~(package-install 'org-ql)~ or
~use-package~ declarations would create a brittle runtime dependency on the
user's ~init.el~.  Vendoring a minimal subset (~org-ql.el~ + 3 deps) keeps
the Emacs bridge self-contained.

The tradeoff: vendored copies can drift from upstream.  Mitigation is that
org-ql's core query engine is stable; the vendored subset excludes the UI
modules (~org-ql-view~, ~org-ql-search~, ~helm-org-ql~) where churn happens.

** Why a TS query builder instead of always delegating to Emacs

Not every deployment has Emacs.  The agent runs on CI machines, containers,
and remote servers where only the Node/Bun runtime is guaranteed.  The TS
query path (~applyFilter~ in ~org-reader.ts~) covers the common predicates
(state, tags, priority, property comparisons) without spawning a subprocess.

Emacs delegation is the high-fidelity path: it handles date math, deadline
warnings, inherited tags, and custom predicates that the TS path cannot.
~requiresEmacs()~ exists precisely to make this routing decision explicit —
callers never silently get degraded results.

The gap: the TS path and Emacs path can disagree on edge cases (tag
inheritance, priority cookie parsing).  The ~query-language.org~ architecture
doc has a predicate fidelity table tracking where they diverge.

** Why text/x-org instead of converting at the protocol boundary

Early design considered converting org → markdown at read time so consumers
wouldn't need to know about org.  This was rejected because:

1. The LLM handles org syntax natively — it's trained on enough org-mode
   content that no conversion is needed for comprehension.
2. Conversion is lossy: PROPERTIES drawers, CLOCK entries, tags, and
   timestamps have no clean markdown equivalent.
3. Converting eagerly would prevent the agent from round-tripping org
   content (read → reason → edit) without accumulating format drift.

The one place conversion matters is the TUI inspector panel, which renders
for human eyes.  That path can call ~orgToMarkdown~ on demand.

** Why SKILL.org takes precedence over SKILL.md

Org frontmatter (~#+KEY: value~) is the native metadata format for org files.
When a skill is authored in org, extracting metadata from org keywords is
more reliable than parsing markdown YAML frontmatter.  Precedence avoids
ambiguity when both files exist (e.g., during migration).

The fallback to ~SKILL.md~ is permanent, not transitional — markdown skills
are a valid authoring choice.  The preference order only applies when both
formats coexist in the same skill directory.

** Why a sessions category with writeInitialPrompt: false

Session items track runtime metadata (start time, duration, agent model,
project context).  Embedding the full system prompt in every session item
would bloat the org files with kilobytes of repeated boilerplate that has
no query value.  ~writeInitialPrompt: false~ keeps session items lean.

The session's system prompt is recoverable from the session log if needed
for debugging; it doesn't need to live in the org item.

* Future Work

** Confidence decay

Memories that are never reinforced should fade.  The proposed mechanism:
on each consolidation pass, entries not referenced in ~raw_memories.md~ since
their ~LAST_VALIDATED~ date have their ~CONFIDENCE~ reduced by a configurable
decay factor.  Entries below a threshold (e.g., 0.1) are archived.

This requires incremental merge (below) as a prerequisite — you can't decay
entries you're overwriting wholesale.

Effort: 4h (after incremental merge).

** Incremental MEMORY.org merge

Current behavior: ~applyConsolidation~ overwrites ~MEMORY.org~ entirely.
Correct behavior: match incoming entries to existing ones by title (or a
stable ~CUSTOM_ID~), update confidence and body, preserve ~CLOCK~ history
and manual edits the user may have made.

This is the single highest-value improvement remaining.  Without it, any
manual curation of ~MEMORY.org~ is lost on the next consolidation.

Effort: 1d.

** Session auto-clocking

On session start: create ~SESS-NNN-<slug>~ item, clock in.  On session end:
clock out, set state to ~DONE~, record final token counts and model info as
properties.  This gives effort-vs-actual tracking for free via org's built-in
~org-clock-report~.

Requires: the agent session lifecycle (~agent-session.ts~) to call the org
tool at start/end boundaries.

Effort: 3h.

** Plan-to-session linking

When ~exit_plan_mode~ creates/approves a plan, the plan's org item should
carry a ~SESSION~ property pointing to the current ~SESS-NNN~ item.  This
enables queries like "show me all plans created in today's sessions" and
builds the cross-reference graph described in ~vision.org~.

Effort: 1h.

** Inspector panel org rendering

The TUI inspector panel (~inspector-panel.ts~) currently does not branch on
~text/x-org~.  When it does, it should call ~orgToMarkdown~ for display
and render the result as markdown.  Low priority — the LLM-facing path
works fine with raw org.

Effort: 2h.

* Problem

The ChatPanel integration tests (=qml-chat-panel.test.ts=) only query the =ListModel= data layer via =messagesModel.get(i)=. They verify model fields (role, text, isStreaming) but never verify that QML delegates actually render those fields into visible =Text= elements.

A bug where:
- =SpellUI.SpellTheme= fails to load (singleton resolution error)
- Delegate =Loader.sourceComponent= selects the wrong =Component= by role
- =model.text= binding breaks inside a delegate context
- Delegate height collapses to zero (=parent= is null guard: =parent ? parent.width : 0=)

...would pass all current tests while producing blank or missing chat bubbles.

The current test architecture has the right layering — a QML test harness wrapper that intercepts bridge messages and supports query/reset protocols. What's missing is queries that walk the visual tree to extract rendered content.

* Approach

Add a =renderedMessages= query to =ChatPanelTestHarness.qml= that walks the =ListView='s instantiated delegate items, finds =Text= children, and returns what's actually rendered. Then add tests that assert rendered text matches expected content.

** How the delegate tree works

#+begin_example
ListView (id: messageList)
  └── delegate: Item
        └── Loader (id: delegateLoader)
              └── sourceComponent: assistantDelegate | userDelegate | toolDelegate
                    └── Rectangle
                          └── Text { text: model.text }  (or ColumnLayout for tool)
#+end_example

The =ListView= exposes =itemAtIndex(index)= (Qt 5.13+) to get the instantiated delegate =Item= at a given model index. From there we walk children recursively to find =Text= elements and extract their =text= property.

QML's =contentItem.children= provides access to all instantiated delegates. However, =ListView= virtualizes — only visible delegates exist. For tests with a few messages this isn't an issue since all items fit in the viewport.

** Visual tree walk function (QML-side)

Add to =ChatPanelTestHarness.qml=:

#+begin_src qml
function collectTexts(item) {
    var texts = []
    if (item.toString().indexOf("QQuickText") !== -1 && item.text !== undefined) {
        texts.push(item.text)
    }
    for (var i = 0; i < item.children.length; i++) {
        texts = texts.concat(collectTexts(item.children[i]))
    }
    return texts
}
#+end_src

The =renderedMessages= query uses this to walk each delegate and extract visible text:

#+begin_src qml
} else if (queryName === "renderedMessages") {
    var rendered = []
    // Force ListView to lay out all items synchronously
    chatPanel.forceLayout()
    var listView = chatPanel.findListView()
    for (var i = 0; i < chatPanel.messagesModel.count; i++) {
        var delegateItem = listView.itemAtIndex(i)
        var entry = {
            index: i,
            exists: delegateItem !== null,
            texts: delegateItem ? collectTexts(delegateItem) : [],
            height: delegateItem ? delegateItem.height : 0,
            visible: delegateItem ? delegateItem.visible : false
        }
        rendered.push(entry)
    }
    result = rendered
}

* Key Design Decisions

** QML-side visual tree walk, not screenshot comparison

Screenshot pixel comparison is fragile across platforms (font rendering differences, Qt software rasterizer variance). Walking the QML visual tree from QML code gives us structural assertions: does the Text element exist, does it contain the expected string, is the delegate instantiated and visible.

** itemAtIndex vs contentItem.children

=ListView.itemAtIndex(index)= (available since Qt 5.13) returns the delegate for a specific model index. This is more reliable than walking =contentItem.children= which includes header/footer items and where ordering doesn't match model indices due to virtualization recycling.

** forceLayout before querying

=ListView= lays out items lazily. After sending messages, the layout may not have updated yet. Calling =forceLayout()= ensures all pending delegate creations and property bindings have completed before we walk the tree.

** Tool details visibility

The tool delegate's detail text (=model.text=) is gated by =model.isExpanded=. The test must either:
- Toggle =isExpanded= before querying (via a new =expandMessage= command)
- Or only assert the tool header (name) which is always visible, and assert the detail text separately via model query

The plan uses option 2 for simplicity: assert the tool name renders in the header, and verify detail text exists in the model. A dedicated =expandMessage= command can be added later if needed.

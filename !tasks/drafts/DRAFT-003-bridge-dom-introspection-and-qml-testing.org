* Problem

The QML bridge (=spell-qml-bridge=) has no DOM introspection capability. The only way to observe QML state from the TS side is through =bridge.send()= messages that QML must explicitly handle — this requires every test wrapper to manually implement query protocols. There's no way to:

- Walk the visual tree to find rendered elements
- Read arbitrary QML properties from C++
- Verify element geometry, visibility, opacity, or layout
- Simulate user input (click, type, scroll)
- Evaluate arbitrary JS expressions in the QML context

The ChatPanel integration tests query the =ListModel= data layer but cannot detect rendering failures: broken delegate bindings, collapsed layouts, invisible elements, missing theme properties.

The QML testing documentation should live in a skill that's surfaced only when the agent is working on QML.

** New bridge commands

*** =query= — DOM tree introspection

Walks the QQuickItem tree from the window root and returns matching items with their properties.

#+begin_src json
{
  "type": "query",
  "id": "test",
  "selector": {
    "type": "QQuickText",
    "objectName": "assistantText",
    "visible": true
  },
  "properties": ["text", "color", "font.pixelSize", "opacity"],
  "includeGeometry": true,
  "recursive": true,
  "maxDepth": 20
}
#+end_src

Response:

#+begin_src json
{
  "type": "query_result",
  "id": "test",
  "items": [
    {
      "className": "QQuickText",
      "objectName": "assistantText",
      "geometry": { "x": 24, "y": 48, "width": 752, "height": 36 },
      "scenePosition": { "x": 24, "y": 48 },
      "visible": true,
      "opacity": 1.0,
      "enabled": true,
      "clip": false,
      "properties": {
        "text": "Hello world",
        "color": "#e6edf3",
        "font.pixelSize": 14,
        "opacity": 1.0
      },
      "childCount": 0,
      "path": "ApplicationWindow/ChatPanel/ColumnLayout/ListView/Item[0]/Loader/Rectangle/Text"
    }
  ]
}
#+end_src

Selector fields (all optional — empty selector matches root):
- =type=: QObject className prefix match (=QQuickText= matches =QQuickText=, =QQuickText_QML_xxxx=)
- =objectName=: exact match on QObject objectName (set via =id:= in QML)
- =visible=: filter by isVisible()
- =property=: ={name, value}= — match items where property equals value
- =text=: shorthand for ={property: {name: "text", value: ...}}= — matches substring

The =properties= array lists QObject property names to read. Dotted paths (=font.pixelSize=) resolve through sub-objects.

*** =eval= — JS expression evaluation

Evaluates a JavaScript expression in the QML engine context with access to the root object.

#+begin_src json
{
  "type": "eval",
  "id": "test",
  "expression": "root.chatPanel.messagesModel.count"
}
#+end_src

Response:

#+begin_src json
{
  "type": "eval_result",
  "id": "test",
  "value": 3,
  "error": null
}
#+end_src

The root object is available as =root= in the expression context. This is the escape hatch for anything the structured query can't express.

** Why both query and eval

=query= handles the common case with structured, typed results and doesn't require the test author to know QML internal IDs. It returns geometry, visibility, and properties in a predictable shape.

=eval= is the escape hatch for:
- Calling QML functions (=chatPanel.messagesModel.clear()=)
- Reading complex nested state
- Triggering interactions (=chatPanel.sendMessage()=)
- Anything the structured query can't express

Both return JSON — no special handling needed on the TS side.

3. =walkTree()= recurses through =item->childItems()= (QQuickItem visual tree, not QObject children — the visual tree is what matters for rendering):

#+begin_src cpp
void WindowManager::walkTree(const QQuickItem *item, const QuerySelector &sel,
                             const QJsonArray &props, bool includeGeometry,
                             int maxDepth, int depth, const QString &path,
                             QJsonArray &results) {
    if (depth > maxDepth) return;

    if (matchesSelector(item, sel)) {
        results.append(serializeItem(item, props, includeGeometry, path));
    }

    const auto children = item->childItems();
    for (int i = 0; i < children.size(); ++i) {
        const auto *child = children[i];
        QString childPath = path + "/" + child->metaObject()->className();
        // Disambiguate siblings of the same type
        int sameTypeCount = 0;
        for (int j = 0; j < i; ++j) {
            if (strcmp(children[j]->metaObject()->className(),
                       child->metaObject()->className()) == 0)
                ++sameTypeCount;
        }
        if (sameTypeCount > 0)
            childPath += "[" + QString::number(sameTypeCount) + "]";
        walkTree(child, sel, props, includeGeometry, maxDepth, depth + 1,
                 childPath, results);
    }
}
#+end_src

4. =matchesSelector()= checks:
   - =type=: =QString(item->metaObject()->className()).startsWith(sel.type)=
   - =objectName=: =item->objectName() == sel.objectName=
   - =visible=: =item->isVisible() == *sel.visible=
   - =textContains=: =item->property("text").toString().contains(sel.textContains)=

5. =serializeItem()= builds a JSON object with:
   - =className=: from metaObject
   - =objectName=: from QObject
   - =geometry=: =x, y, width, height= from QQuickItem
   - =scenePosition=: =mapToScene(QPointF(0,0))=
   - =visible=, =opacity=, =enabled=, =clip= from QQuickItem
   - =properties=: each requested property read via =readProperty()=
   - =childCount=: =childItems().size()=
   - =path=: the tree path string

6. =readProperty()= handles dotted paths:
   - Split on =.=
   - First segment: =obj->property(name)=
   - If QVariant wraps a QObject*, recurse to next segment
   - If QVariant wraps a QFont/QColor, use specific extraction

#+begin_src cpp
void WindowManager::evalInWindow(const QString &id, const QString &expression) {
    auto it = m_windows.find(id);
    if (it == m_windows.end()) { /* error event */ return; }

    QObject *root = it->engine->rootObjects().first();
    // Create a QJSValue for 'root' and evaluate expression with it
    QJSEngine *jsEngine = it->engine;  // QQmlApplicationEngine IS-A QJSEngine
    QJSValue rootVal = jsEngine->newQObject(root);
    QJSValue globalObj = jsEngine->globalObject();
    globalObj.setProperty("root", rootVal);

    QJSValue result = jsEngine->evaluate(expression);
    globalObj.deleteProperty("root"); // cleanup

    QJsonObject ev;
    ev["type"] = "eval_result";
    ev["id"] = id;
    if (result.isError()) {
        ev["error"] = result.toString();
        ev["value"] = QJsonValue::Null;
    } else {
        ev["error"] = QJsonValue::Null;
        ev["value"] = result.toVariant().toJsonValue();
    }
    writeEvent(ev);
}
#+end_src

Notes:
- =QQmlApplicationEngine= inherits =QJSEngine= so we can evaluate JS directly
- We inject =root= into the global object temporarily, then clean up
- Result serializes through =QJSValue::toVariant().toJsonValue()= — handles numbers, strings, booleans, arrays, objects
- QObject references in results would serialize as empty objects; this is acceptable since the primary use is reading values, not references

* Key Design Decisions

** Bridge-level, not QML-level

DOM introspection lives in C++ (=windowmanager.cpp=), not in per-test QML wrappers. Advantages:
- Every QML window gets it for free — no test wrapper boilerplate
- Structured query language instead of ad-hoc QML functions
- Property reading works for any QQuickItem subclass without QML-side code
- The =eval= escape hatch covers anything the structured query misses

** =query= for structure, =eval= for escape

The =query= command handles the 90% case (find elements, read properties, check geometry) with a structured selector+response format. The =eval= command handles everything else by running arbitrary JS in the QML engine. Tests should prefer =query= for readability and type safety, using =eval= only when needed.

** Prefix matching for type selectors

QML-generated types have mangled names (=QQuickText= vs =QQuickText_QMLTYPE_42=). Using =className().startsWith(sel.type)= handles both. The caller writes =QQuickText=, it matches all variants.

** Skill for documentation, not code

The =qml-testing= skill is a documentation artifact, not code. It teaches the agent how to use the bridge's test capabilities. The description is concise enough to appear in every system prompt without waste, but detailed enough that the agent knows when to read the full content.

** No QML-side =objectName= requirements

The query command works without =objectName= by using type + property matching. Tests shouldn't require modifying production QML to add objectNames — the tree structure and property values are sufficient for assertions.

* Risk Mitigation

** Qt version compatibility

=QQuickItem::childItems()= and =QObject::property()= exist since Qt 5.0. =itemAtIndex()= (on ListView) is Qt 5.13+. The query command uses =childItems()= which is universally available. No version risk.

** Performance of tree walk

A typical ChatPanel with 20 messages has ~200 items in the visual tree. Walking this is sub-millisecond. The =maxDepth= parameter prevents runaway recursion on pathological trees. No performance risk.

** eval security

The =eval= command runs in the QML engine's JS context, which is sandboxed from the filesystem. It can only access what the QML context exposes. Since the bridge is already a local process controlled by spell, there's no privilege escalation concern.

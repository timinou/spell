* Problem

The TUI is a fixed, terminal-bound interface. The user wants Spell to boot into a QML desktop UI where:
1. The agent can modify the QML files composing its own UI (self-modifying)
2. QML hot-reloads without restarting the session
3. Skills ship saved QML interfaces as discoverable panels
4. A shell QML (MQML) manages loading child QML views safely
5. Bad QML from the agent must not crash Spell
6. Agent/tool code hot-reload via process restart — the QML bridge runs as a *daemon* that survives Spell restarts
7. Visual style inspired by DankMaterialShell's Material Design 3 approach

* Existing Infrastructure

** Agent runtime is decoupled from TUI

~AgentSession~ exposes: ~subscribe(listener)~ for events, ~prompt(text)~ for input, ~abort()~, ~steer()~. The TUI (~interactive-mode.ts~) is just one consumer. ~print-mode.ts~ (150 lines) shows the minimal driver pattern. ~rpc-mode.ts~ shows full programmatic control via JSON-lines.

** QML bridge is production-ready (but not daemonized)

~packages/qml/~ provides:
- C++ subprocess (~spell-qml-bridge~) using stdio (JSON-newline on stdin/stdout)
- ~QmlBridge~ TS class: launch/reload/close/sendMessage/screenshot
- ~QmlWatcher~: fs.watch with 150ms debounce for hot-reload
- Error events forwarded to TS when QML fails to load

Current limitation: ~QmlProcess~ spawns the bridge as a child process via ~Bun.spawn~. It dies when Spell dies. For persistent UI across agent restarts, this must become a daemon that listens on a Unix domain socket.

** Session persistence exists

~SessionManager~ persists entries as JSONL files. ~--resume~ / ~--session~ flags restore sessions via ~setSessionFile()~ which loads entries and rebuilds state. This is the continuity mechanism across restarts.

** Mode dispatch in main.ts

Lines 712-757: branches on mode — \"rpc\", interactive, print. Adding \"qml\" follows the same pattern.

** Existing skill QML

~.spell/skills/brand-asset-gallery/gallery.qml~ (402 lines) shows the current pattern — a full ApplicationWindow with bridge.props for data and bridge.send for events.

* Architecture

** Two-layer QML structure

*** Shell QML (MQML)

Root ApplicationWindow acting as compositor/shell:
- Sidebar/drawer listing available panels (skills, agent-created, user-saved)
- Loads child panels into a ~Loader~ component
- Built-in chat panel (always available) for prompting the agent
- Catches child panel load errors via ~Loader.status === Loader.Error~ — shows error text + Retry, keeps running
- Routes ~bridge.messageReceived~ events to active panel
- \"Restart Agent\" button to trigger Spell process restart while keeping QML alive

*** Panel QML (views)

Individual .qml files loaded by the shell via ~Loader { source: panelPath }~. Not full ApplicationWindows — they are Items loaded into the shell's content area.

** Daemon QML bridge

                    /tmp/spell-qml-$UID.sock
                         |
    spell --qml  <------>|<------->  spell-qml-bridge (daemon)
    (may restart)        |           (persistent, owns all windows)
                         |
    spell --qml  <------>|  (reconnects after restart)
    (new process)
#+end_example

*** C++ changes to spell-qml-bridge

Replace stdin/stdout with ~QLocalServer~ listening on a Unix domain socket:
- Socket path: ~/tmp/spell-qml-bridge-$UID.sock~ (or ~$XDG_RUNTIME_DIR/spell-qml-bridge.sock~)
- Accept one connection at a time (single-client model)
- On client disconnect: keep windows alive, wait for reconnection
- On reconnect: send ~{type: \"state\", windows: [...]}~ listing all active windows so the new Spell process can rebuild its ~QmlBridge~ window map
- ~writeEvent()~ writes to connected socket instead of stdout
- ~dispatch()~ reads from socket instead of stdin
- Startup: if socket already exists and a bridge is running, the new bridge exits (only one daemon)
- Shutdown: explicit ~{type: \"quit\"}~ command, or when no client has connected for a configurable timeout

*** TypeScript changes to QmlProcess

- ~ensure()~: try connecting to existing socket first (~Bun.connect~ or ~net.createConnection~). If no daemon running, spawn it, then connect.
- ~dispose()~: disconnect from socket but do NOT kill the daemon
- Add ~kill()~ method for explicit daemon shutdown
- Read/write via socket instead of process stdio
- On connection loss (daemon died): respawn and reconnect, same as current respawn logic

** Process restart flow

#+begin_example
User clicks \"Restart Agent\" in QML shell
    |
    v
QML: bridge.send({type: \"restart\"})
    |
    v
qml-mode.ts:
    1. Records current session file path
    2. session.dispose() — flushes JSONL, does NOT kill QML daemon
    3. Spawns new Spell: Bun.spawn([\"spell\", \"--qml\", \"--resume\", sessionFile])
    4. Current process exits
    |
    v
New Spell process:
    1. QmlProcess.ensure() → connects to existing daemon socket
    2. Daemon sends state snapshot (active windows)
    3. QmlMode rebuilds window map from snapshot
    4. createAgentSession() with --resume → restores messages
    5. Subscribes to session events → resumes forwarding to QML
    6. Session continues seamlessly — QML shell never flickered
#+end_example

** Event protocol (QML <-> Agent)

AgentSessionEvent -> QML (via bridge.sendMessage):
- ~message_start~ → ~{ type: \"message_start\", id }~
- ~message_update~ → ~{ type: \"message_update\", id, text, thinking }~
- ~message_end~ → ~{ type: \"message_end\", id }~
- ~tool_execution_start~ → ~{ type: \"tool_start\", id, name, input }~
- ~tool_execution_update~ → ~{ type: \"tool_update\", id, partialResult }~
- ~tool_execution_end~ → ~{ type: \"tool_end\", id, result, error }~
- ~agent_start/end~ → ~{ type: \"agent_busy\", busy: bool }~

QML -> Agent (via bridge.send):
- ~{ type: \"prompt\", text }~ → ~session.prompt(text)~
- ~{ type: \"abort\" }~ → ~session.abort()~
- ~{ type: \"steer\", text }~ → ~agent.steer(text)~
- ~{ type: \"restart\" }~ → re-exec Spell (daemon stays alive)

* Style Direction (DMS-Inspired)

** Design philosophy

Take inspiration from DankMaterialShell's approach to Material Design 3:
- *Semantic color tokens* not raw hex — components reference roles (~surface~, ~onSurface~, ~primary~, ~container~, ~outline~) not literal colors
- *State layers* — hover (0.08 opacity), pressed (0.12 opacity), disabled (0.38 opacity) applied as overlays, exactly as DMS's ~StateLayer.qml~
- *Smooth transitions* — all color/opacity changes animated with standard easing curves and short durations (~150-200ms~)
- *Elevation through tonal surfaces* — not box-shadows. Surface1 < Surface2 < Surface3 each slightly brighter/lighter to convey depth
- *Generous spacing* — DMS uses a spacing scale (~spacingXS~, ~spacingS~, ~spacingM~, ~spacingL~, ~spacingXL~). Do the same.
- *Consistent corner radii* — single ~cornerRadius~ token, larger for containers, smaller for chips/badges
- *Typography scale* — ~fontSizeSmall~, ~fontSizeMedium~, ~fontSizeLarge~; mono font for code blocks; \"Inter\" or similar variable font for UI text

** Theme singleton

** Component patterns (inspired by DMS widgets)

*** StateLayer

Copy DMS's pattern exactly — a MouseArea that fills parent, applies tonal overlay on hover/press:

#+begin_example
Rectangle {
    anchors.fill: parent
    radius: parent.radius
    color: SpellTheme.withAlpha(
        SpellTheme.onSurface,
        pressed ? SpellTheme.pressOpacity
                : hovered ? SpellTheme.hoverOpacity : 0
    )
    Behavior on color { ColorAnimation { duration: SpellTheme.shortDuration } }
}
#+end_example

*** Cards / containers

DMS-style: no border by default, tonal surface differentiation, subtle outline only on focus/active:

#+begin_example
Rectangle {
    color: SpellTheme.surface
    radius: SpellTheme.cornerRadius
    border.width: isActive ? 1 : 0
    border.color: SpellTheme.outline
}
#+end_example

*** Chat bubbles

Agent messages: ~surfaceHigh~ background, full width
User messages: ~primaryContainer~ background, right-aligned, rounded

*** Tool call cards

Collapsible, with icon + tool name header in ~onSurfaceDim~, expand to show input/output.
Status indicator: spinning for active, checkmark for complete, X for error.

*** Code blocks

~surfaceHigher~ background, ~monoFontFamily~, ~cornerRadiusSmall~ radius, left border accent in ~primary~.

*** Scrollbar

Thin, rounded, ~onSurfaceFaint~ colored, auto-hide with opacity animation. DMS's ~DankScrollbar.qml~ pattern.

** What NOT to copy from DMS

- No matugen/dynamic wallpaper theming — Spell is not a desktop shell
- No Quickshell-specific imports (~Singleton~, ~Variants~, ~PanelWindow~) — we use plain Qt6
- No system service singletons — our data comes from bridge.props/messageReceived
- Keep it simpler — DMS is 100K+ lines of QML. Our shell should be <2000 lines initially.

#+begin_src typescript
async restart(): Promise<never> {
  const sessionFile = this.sessionManager.sessionFile;
  await this.session.dispose(); // flushes JSONL, disconnects from daemon
  const args = process.argv.slice(1).filter(a => a !== \"--resume\" && !a.startsWith(\"--session\"));
  args.push(\"--resume\", sessionFile);
  const proc = Bun.spawn([process.argv[0], ...args], {
    stdio: [\"inherit\", \"inherit\", \"inherit\"],
    env: process.env,
  });
  process.exit(await proc.exited);
}
#+end_src

* Problem

=/plan= works well but produces plans without quality gates. The agent writes a draft, calls =exit_plan_mode=, and the user approves. There's no automated gap analysis (did the agent miss requirements?) or validation (is the plan concrete enough to execute without guessing?).

Inspired by oh-my-opencode's Planning Triad (Prometheus/Metis/Momus), we want =/ultraplan= — an enhanced planning mode where:
- *Metis* (mandatory): read-only gap analysis subagent runs before the plan is written, catching implicit assumptions, missing edge cases, AI-slop patterns, and ambiguous requirements
- *Momus* (optional, agent asks user): read-only plan reviewer that validates the written draft against quality thresholds (file references verified, acceptance criteria concrete, zero business logic assumptions)

=/plan= stays unchanged. =/ultraplan= adds these quality gates.

* Design

** Architecture

The implementation is prompt-layer + wiring. No new tool types, no new agent runtime features. We add:
1. Two new bundled agent definitions (=metis.md=, =momus.md=)
2. A new slash command =/ultraplan=
3. Extended =PlanModeState= with an =ultraplan= flag
4. Updated =plan-mode-active.md= prompt template to instruct the agent on the Metis/Momus workflow when ultraplan is active

The key insight: Metis and Momus don't need special runtime support. They're just =task= tool subagents the planning agent invokes during its work, guided by the system prompt. The system prompt for ultraplan mode instructs the agent:
1. Spawn Metis subagent with the user's requirements + codebase context
2. Incorporate Metis findings silently into the plan
3. Write the DRAFT org item
4. Ask the user if they want high-accuracy review (Momus)
5. If yes, spawn Momus subagent with the draft item path
6. If Momus rejects, revise and resubmit (loop)
7. Call =exit_plan_mode=

** Key Decision: Prompt-driven vs Code-driven orchestration

Prompt-driven. The agent already has =task= tool access in plan mode (for spawning =explore= agents). We add Metis/Momus as agent definitions the planning agent can spawn. The =plan-mode-active.md= template gets an ={{#if ultraplan}}= section with the orchestration instructions.

This avoids new code paths in =interactive-mode.ts= or =agent-session.ts=. The orchestration lives entirely in prompt + agent definitions.

- spawns: explore
- model: pi/slow

Prompt body instructs Metis to:
- Analyze the user's requirements and codebase context provided as input
- Identify hidden intentions, ambiguities, missing acceptance criteria, edge cases
- Flag AI-slop patterns: over-engineering, scope creep, unnecessary abstractions
- Ensure all acceptance criteria are agent-executable (no manual testing required)
- Return structured output: gaps found, risks, recommendations, missing context

Key principle: Metis is read-only. It cannot write files or delegate to other task agents.

** ITEM Create =momus.md= bundled agent
:PROPERTIES:
:EFFORT: 1h
:END:

File: =packages/coding-agent/src/prompts/agents/momus.md=

Frontmatter:
- name: momus
- description: Plan quality reviewer. Validates plans against clarity, verification, context, and completeness criteria. Returns APPROVE or REJECT with specific issues.
- tools: read, grep, find, bash, lsp, ast_grep
- model: pi/slow
- thinking-level: high
- blocking: true

Prompt body instructs Momus to:
- Read the plan (provided as input — org item body or file path)
- Validate against four criteria:
  1. *Clarity*: each task specifies WHERE (file paths, patterns, line ranges)
  2. *Verification*: acceptance criteria are concrete, measurable, agent-executable
  3. *Context*: sufficient context to proceed with <10% guesswork
  4. *Completeness*: purpose, background, workflow documented
- Quality thresholds:
  - File references: 100% must exist in codebase
  - Reference sources: >=80% of tasks have clear patterns/APIs/examples
  - Acceptance criteria: >=90% of tasks have concrete verification
  - Business logic assumptions: zero allowed
- Return APPROVE or REJECT with specific issues
- Approval bias: reject only for true blockers, not perfectionism

Key principle: Momus is read-only. Named after the Greek god of criticism — ruthless but fair.

* Risks

- *Token cost*: Metis + Momus are expensive subagent calls (high thinking, slow model). The user should understand =/ultraplan= costs ~2-3x a normal =/plan=.
- *Prompt compliance*: The orchestration is prompt-driven. If the agent doesn't follow the instructions (skips Metis, forgets to ask about Momus), there's no code enforcement. This is acceptable — the same pattern works in oh-my-opencode.
- *Momus loop*: No maximum retry limit. If Momus keeps rejecting, the agent loops. The user can always exit plan mode manually (=Alt+Shift+P= or =/plan= to toggle off).

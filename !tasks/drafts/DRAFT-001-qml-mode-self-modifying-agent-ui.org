* Initial Prompt
* Problem

The TUI is a fixed interface. The user wants Spell to boot into a QML-based desktop UI where:
1. The agent can modify the QML files that compose the UI it's running in (self-modifying)
2. QML hot-reloads without restarting the session
3. Skills can ship saved QML interfaces (panels) that appear as presets
4. The shell QML (MQML) manages loading/unloading child QML views safely
5. Bad QML written by the agent must not crash Spell

Design inspiration: Quickshell (~/.config/quickshell) — modular QML composition.

* Architecture

** Two-layer QML structure

*** Shell QML (MQML)
A root ApplicationWindow that acts as a compositor/shell. It:
- Provides a sidebar/drawer listing available panels (from skills, agent-created, or user-saved)
- Loads child QML views into a StackView or Loader
- Has a built-in chat panel (always available) for prompting the agent
- Catches QML load errors from child panels and shows an error view instead of crashing
- Receives all AgentSessionEvents via bridge.messageReceived and dispatches to active panels

*** Panel QML (views)
Individual .qml files that render specific UIs — chat, file diffs, brand gallery, etc.
Loaded dynamically by the shell via Loader { source: panelPath }.
If a panel fails to load, the shell catches the error and shows fallback UI.

** Error isolation for agent-written QML

The C++ bridge currently crashes or emits error when QML fails to load.
For self-modifying UI, we need:
1. The MQML shell uses QML Loader with error handling (Loader.status === Loader.Error)
2. On error: show the error message in a panel, keep the shell alive
3. The agent sees the error via bridge events and can fix the QML
4. This is handled entirely in QML-land (no C++ changes needed) — Loader already supports this

** Skill QML discovery

Skills already live in .spell/skills/<name>/. Discovery adds:
- Glob for *.qml in each skill directory
- Expose found panels as { skillName, qmlPath, title } to the MQML shell via bridge.props
- Example: brand-asset-gallery/gallery.qml becomes a launchable panel

* Implementation Plan

** ITEM Create qml-mode.ts
:PROPERTIES:
:EFFORT: 4h
:END:

New mode driver at ~packages/coding-agent/src/modes/qml-mode.ts~.

Pattern follows print-mode.ts (simplest existing mode):
- Accept AgentSession
- Launch QML shell via QmlBridge
- Subscribe to AgentSession events, forward as JSON to QML via bridge.sendMessage
- Listen for bridge events from QML, dispatch to session.prompt/abort/steer
- Handle graceful shutdown

Event mapping (AgentSessionEvent → QML message):
- message_start/update/end → streaming text chunks
- tool_execution_start/update/end → tool call status + results
- agent_start/end → busy indicator
- auto_compaction → compaction notice

QML → Agent mapping (bridge.send → session method):
- { type: "prompt", text } → session.prompt(text)
- { type: "abort" } → session.abort()
- { type: "steer", text } → agent.steer(text)

** ITEM Agent self-modification safety
:PROPERTIES:
:EFFORT: 2h
:END:

When agent edits a .qml file that's currently loaded:
1. QmlWatcher triggers reload (already works)
2. If reload fails, MQML Loader catches the error, shows it
3. Agent receives error event via bridge → can fix the QML
4. Need: a "validate QML" step before reload — either:
   a. Use QQmlEngine::loadFromData in a try-catch in C++ (heavyweight)
   b. Just let it fail and show the error (simpler, sufficient)
   
Choose (b): fail-and-show. The agent is fast enough to iterate.

Add to the qml tool: when writing a .qml file, if a window is using it, the tool result should include any reload errors so the agent sees them without polling.

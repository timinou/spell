File: =packages/coding-agent/src/modes/qml/panels/ChatPanel.qml=

The Loader→Component pattern may break model property change notification propagation. Two options:

*Option A (preferred):* Instead of Components loaded via Loader, use inline delegates or use a =DelegateChooser= (Qt 5.12+). Simplest fix: replace the Loader+Component pattern with a single delegate that shows/hides sections based on =model.role=:

#+begin_src qml
delegate: Item {
    width: messageList.width - messageList.leftMargin - messageList.rightMargin
    height: childrenRect.height

    // Assistant bubble
    Rectangle {
        visible: model.role === "assistant"
        width: parent.width
        height: visible ? aText.paintedHeight + SpellUI.SpellTheme.spacingL * 2 : 0
        color: SpellUI.SpellTheme.surfaceHigh
        radius: SpellUI.SpellTheme.cornerRadius

        Text {
            id: aText
            anchors.fill: parent
            anchors.margins: SpellUI.SpellTheme.spacingL
            text: model.text
            // ... font/color/wrap properties
        }
    }

    // User bubble (right-aligned)
    Item {
        visible: model.role === "user"
        // ... same structure as current userDelegate
    }

    // Tool entry
    Rectangle {
        visible: model.role === "tool"
        // ... same structure as current toolDelegate
    }
}
#+end_src

This ensures =model.text= binding is direct within the ListView delegate context, where ListModel change notifications always propagate.

case "tool_execution_end":

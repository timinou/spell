* Problem

Two issues:

1. The org tool treats Emacs as optional — every command that needs Emacs has a TS fallback or a null-guard. Emacs is always available, so this is dead code.

2. The ~query~ command always uses the TS-side ~applyFilter~ path. The org-ql query engine exists (elisp tool registered, sexp builder implemented, ~requiresEmacs~ classifier ready) but is never called. The agent already uses keyword syntax (~todo:DOING tags:auth priority:>=B~) which ~parseKeywordQuery~ converts to ~OrgQlFilter~, and ~buildOrgQlSexp~ can convert that to org-ql sexps. The wiring is missing.

* Part 2: Wire org-ql into query path

** ITEM Add OrgClient to OrgContext (~packages/org/src/tool.ts~)

~OrgContext~ gains a lazy ~getOrgClient()~ method that creates an ~OrgClient~ from the session's socket path (via ~createOrgClient~). Created once, cached.

#+begin_src typescript
interface OrgContext {
  config: OrgConfig;
  projectRoot: string;
  getEmacsSession(): Promise<EmacsSession>;
  getOrgClient(): Promise<OrgClient>;
  getSessionContext?(): OrgSessionContext;
}
#+end_src

In ~createOrgTool~, wire it:
#+begin_src typescript
let orgClientPromise: Promise<OrgClient> | null = null;
const ctx: OrgContext = {
  // ...
  async getOrgClient() {
    if (!orgClientPromise) {
      orgClientPromise = ctx.getEmacsSession().then(s =>
        createOrgClient(s.socketPath)
      );
    }
    return orgClientPromise;
  },
};
#+end_src

Note: ~createOrgClient~ currently returns ~OrgClient | null~ (null if socat missing). Since Emacs is always available, this should also be made non-nullable (socat is checked during detection).

The agent uses the same keyword syntax. Simple queries (~todo:DOING priority:>=B~) stay TS-side. Advanced queries (~clocked~, ~dateRange~, ~effort~, numeric property ops) transparently route to org-ql. No org-ql syntax exposed to the agent.

** ITEM Add org-ql sexp passthrough parameter (optional, for power users)

Add an optional ~ql~ parameter to the query schema — a raw org-ql sexp string. When provided, it bypasses keyword parsing and goes straight to the Emacs org-ql-query tool. This is for internal/advanced use only; the tool description doesn't advertise it prominently.

This keeps the door open for internal subsystems (memory consolidation, planning) to issue precise org-ql queries without going through keyword syntax.

* Part 3: Clean up architecture docs

** ITEM Update query-language.org

Remove the "dual-path" framing that assumes Emacs may not be available. The TS path is a fast path for simple queries, not a fallback. Reframe:

- Raw org-ql: optional ~ql~ parameter for power users / internal subsystems

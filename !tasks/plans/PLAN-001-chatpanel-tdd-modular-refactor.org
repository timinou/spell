* Problem

ChatPanel.qml is a 435-line monolith: three inline Component delegates, an
if/else-if message dispatcher, a plain-text renderer that will never grow
into markdown, and no armed-tool integration. The user wants a *chef's kiss*
modular architecture that anticipates the markdown renderer growing richer
over time — every rendering concern in its own file, composable, testable
in isolation.

The QML test harness (QmlTestHarness) now has deep capabilities:
~findItems()~ with type/objectName/textContains selectors, ~findVisibleText()~,
~assertVisible()~, ~assertNotFound()~, ~evaluate()~ for arbitrary JS in QML
context, plus the existing ~query()~/~sendMessage()~/~screenshot()~ protocol.
These must be leveraged throughout.

* Architecture: Component Tree

The target structure for ~packages/coding-agent/src/modes/qml/panels/~:

#+begin_src
panels/
  ChatPanel.qml              — orchestrator: ListView + InputBar + StatusBar
  InputBar.qml               — text input + send/abort buttons
  StatusBar.qml              — model name + token count
  delegates/
    MessageDelegate.qml      — Loader dispatcher: role → component
    AssistantBubble.qml       — wraps MarkdownContent
    UserBubble.qml            — right-aligned bubble, plain text
    ToolCard.qml              — header (name + status icon) + collapsible body + progress
  markdown/
    MarkdownContent.qml       — Column: RichText prose + CodeBlock items
    CodeBlock.qml             — fenced code: language header + monospace body + copy/save buttons
    MarkdownParser.mjs        — JS module: markdownToSegments(text) → [{type, content, lang?}]
  ChatPanelTestHarness.qml   — test wrapper (extended for new queries)
#+end_src

Key design principles:
- *One component, one concern.* Each .qml file does exactly one thing.
- *Data flows down via required properties.* Components declare what they need;
  the Loader passes it via ~setSource(url, {props})~. No reaching up via ~parent~.
- *Events flow up via signals/bridge.send.* Components never reference siblings.
- *MarkdownParser is pure JS.* No QML imports — testable from TS via ~evaluate()~.
- *Markdown rendering is a Column with O(code_blocks) children, not O(all_blocks).*
  Prose is a single RichText Text item; only code blocks get their own Rectangle
  (for the copy/save button). This scales during streaming.

* Approach: Strict TDD Phases

Each phase is green-to-green: all existing tests pass before AND after.
New behavior requires a failing test first.

*** Step 1a: Extract MessageDelegate.qml

Current pattern: inline ~delegate: Item { Loader { sourceComponent: ... } }~
Target pattern: ~delegate: MessageDelegate { ... }~

MessageDelegate.qml becomes the dispatcher file. It receives model data via
required properties from the ListView delegate context and uses
~Loader.setSource()~ to instantiate the right sub-component.

#+begin_src qml
// MessageDelegate.qml
Item {
    id: root
    required property string msgId
    required property string role
    required property string text
    required property string name
    required property bool isStreaming
    required property bool isExpanded
    required property int index

    width: ListView.view ? ListView.view.width - ListView.view.leftMargin - ListView.view.rightMargin : 0
    height: delegateLoader.item ? delegateLoader.item.height : 0

    Loader {
        id: delegateLoader
        width: parent.width
        Component.onCompleted: {
            var url = role === "user" ? "delegates/UserBubble.qml"
                    : role === "tool" ? "delegates/ToolCard.qml"
                    : "delegates/AssistantBubble.qml"
            setSource(Qt.resolvedUrl(url), {
                text: Qt.binding(function() { return root.text }),
                name: Qt.binding(function() { return root.name }),
                isStreaming: Qt.binding(function() { return root.isStreaming }),
                isExpanded: Qt.binding(function() { return root.isExpanded }),
                messageIndex: root.index
            })
        }
    }
}
#+end_src

*** Step 2a: MarkdownParser.mjs — pure JS, TDD via evaluate()

A JS module that converts markdown text to a segment array:

#+begin_src js
// Input: "Hello **bold** and\n```js\nconst x = 1\n```\nMore text"
// Output:
[
  { type: "rich", html: "Hello <b>bold</b> and" },
  { type: "code", lang: "js", content: "const x = 1" },
  { type: "rich", html: "More text" }
]
#+end_src

Supported markdown (maps to Qt RichText HTML subset):
- Bold (**text**) → ~<b>~
- Italic (*text*) → ~<i>~
- Inline code (`text`) → ~<code>~ with monospace font styling
- Fenced code blocks (```lang ... ```) → separate ~code~ segments
- Unordered lists (- item) → ~<ul><li>~
- Ordered lists (1. item) → ~<ol><li>~
- Headers (# ## ###) → styled ~<span>~ with size
- Links [text](url) → ~<a href>~
- Line breaks → ~<br>~
- Horizontal rules (---) → separate ~hr~ segment

*Streaming edge case*: unclosed code fences treated as open code block
(the fence will close on the next update). The parser is stateless and
re-invoked on every text change — this is cheap since it produces a flat
segment array, not QML items.

*Tests* (all via harness.evaluate(), no QML rendering needed):

#+begin_src typescript
it("parses plain text as single rich segment", async () => {
    const result = await harness.evaluate<Segment[]>(
        `Qt.include("markdown/MarkdownParser.mjs"); markdownToSegments("Hello world")`
    );
    expect(result).toEqual([{ type: "rich", html: "Hello world" }]);
});

it("extracts fenced code block with language", async () => {
    const result = await harness.evaluate<Segment[]>(
        `markdownToSegments("before\\n\`\`\`js\\nconst x = 1\\n\`\`\`\\nafter")`
    );
    expect(result).toHaveLength(3);
    expect(result[1]).toEqual({ type: "code", lang: "js", content: "const x = 1" });
});

it("handles unclosed code fence during streaming", async () => {
    const result = await harness.evaluate<Segment[]>(
        `markdownToSegments("text\\n\`\`\`python\\nprint('hi')")`
    );
    expect(result[1].type).toBe("code");
    expect(result[1].content).toBe("print('hi')");
});

#+begin_src qml
Item {
    required property string text

*** waitForBridgeEvent(predicate, timeout)

Thin wrapper over ~#process.waitFor()~ that waits for a raw bridge event
matching a predicate. Used to capture outgoing ~_tool~ invocations from QML.

#+begin_src typescript
async waitForBridgeEvent(
    predicate: (event: BridgeEvent) => boolean,
    timeout = 5_000
): Promise<BridgeEvent> {
    if (!this.#process) throw new Error("not set up");
    return this.#process.waitFor(predicate, timeout);
}
#+end_src

*** simulateToolResponse(rid, payload)

Convenience for simulating the _rid response round-trip:

#+begin_src typescript
async simulateToolResponse(rid: string, payload: Record<string, unknown>): Promise<void> {
    await this.sendMessage({ _rid: rid, ...payload });
}
#+end_src

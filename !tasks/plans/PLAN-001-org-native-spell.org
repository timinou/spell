* The Insight

Spell already has two parallel document systems running side by side. The org subsystem (packages/org/) is surprisingly mature — 12 subcommands, Emacs daemon bridge with clocking/agenda/dependency graphs, plan-mode integration, journal serialization. But it's walled off. Every other system — memory, skills, artifacts, plans, protocol handlers — speaks markdown exclusively. The org tool creates structured data that immediately gets labeled =text/markdown= and rendered through =marked.lexer()=.

The result: Spell's knowledge management is a flat bag of prose. Memories have no queryable metadata. Plans can't track actual effort. Skills can't be filtered by domain. Sessions leave no structured trace. The org tool is an island of structure in an ocean of unstructured text.

Making Spell org-native isn't about file format preference. It's about giving the agent structured, queryable state that survives across sessions.

* What Org Enables That Markdown Cannot

** Queryable Memory with Confidence Decay

Current memory: free-form markdown blobs. No structure, no metadata, no way to ask "what do I know about auth with high confidence?" Consolidation rewrites the entire corpus each time.

Org-native memory: each memory is a heading with a PROPERTIES drawer.

#+begin_example
,* JWT token validation
:PROPERTIES:
:CONFIDENCE: 0.9
:LAST_VALIDATED: [2026-03-10]
:SCOPE: backend/auth
:SOURCE_SESSION: abc123
:END:
Use RS256 not HS256. Validation middleware at src/auth/validate.ts.
#+end_example

This enables:
- *Property-based filtering at read time*: inject only memories with CONFIDENCE > 0.7 relevant to current scope
- *Incremental consolidation*: append new memories, archive stale ones, adjust scores — no full rewrite
- *Confidence decay*: memories unvalidated for 30 days get lower injection priority
- *Source tracing*: every memory links to the session that produced it

** Automatic Effort Tracking on Plans

The Emacs bridge already has =org-clock-in= / =org-clock-out= wired. If sessions clock in on start and out on end, every plan accumulates a LOGBOOK. "Estimated 6h, actual 9h across 4 sessions" — tracked automatically.

** Session-as-Org-Item

Every coding session becomes a queryable org item. "All sessions touching auth in the last week" becomes an org-query, not a grep through markdown blobs.

** Cross-Session Knowledge Graphs

If plans, sessions, and memories are all org items with cross-links via CUSTOM_ID, the knowledge graph the memory system approximates with prose becomes navigable structure.

** Emacs as First-Class Spell UI

If all Spell state lives in org files, Emacs users interact with state directly via org-agenda. Spell becomes a "headless org-mode agent" that Emacs users supervise natively.

** Unified Query Across All Spell State (via org-ql)

org-ql ([[https://github.com/alphapapa/org-ql]]) provides a full query language for Org files — Lisp sexps and keyword syntax. Vendored into the Emacs daemon:

| Spell Domain | org-ql Predicates |
|--------------+-------------------|
| Memories     | =(property "SCOPE" "backend/auth")=, =(ts :from -30)=, custom =confidence-above= predicate |
| Plans        | =(todo "DOING" "ITEM")=, =(priority >= "B")=, =(effort <= "4h")=, =(clocked :on today)= |
| Sessions     | =(clocked :on today)=, =(ts :from -7)=, =(property "PROJECT" "spell")=, =(tags "auth")= |
| Skills       | =(property "DOMAIN" "typescript")=, =(tags "testing")= |

Note: org-ql's =(property)= does string matching only. Numeric CONFIDENCE comparison needs a custom Emacs-side predicate or TS-side post-filter.

* What NOT to Change

- *System prompt templates (.md)*: LLMs are trained on markdown. Prompts stay .md.
- *Subagent artifact output*: LLMs produce markdown. Agent output stays .md.
- *TUI core rendering*: Keep =marked.lexer()=. Org content goes through =orgToMarkdown()= facade (uniorg pipeline) upstream.

Principle: *org for structured data with queryable metadata; markdown for prose consumed by LLMs*.

* Current State of Org in Spell

What exists today:
- =packages/org/= — pure-TS line-by-line parser for CRUD, Emacs daemon bridge (13 elisp tools), categories, ID generation
- Plan mode branches on =orgEnabled= — creates org items with EFFORT/PRIORITY/LAYER
- Journal serializes todo phases to =.org= files
- Emacs bridge has clock-in/out, agenda, dependency graph, validation
- =org://= protocol handler (returns org content labeled =text/markdown= — a type lie)
- Emacs vendoring pattern: =packages/emacs/elisp/vendor/= has combobulate/, mcp/, tree-sitter/

What's broken:
- =contentType= union excludes org (=text/markdown | application/json | text/plain= only)
- =org://= returns org body as =text/markdown=
- TS parser handles headings + PROPERTIES only — no inline markup, links, tables, timestamps
- Plan mode enforces =.md= extension via regex
- Memory, skills all hardcoded to markdown

* Risks

** LLM Output Quality for Org Syntax
Mitigation: LLM-facing content stays markdown. The LLM never produces org — the org tool handles serialization.

** Ecosystem Friction
Mitigation: Offer org-to-md export at boundaries. Canonical format org, generate .md where external tools need it.

** Parser Fidelity
Mitigation: Adopt [[https://github.com/rasendubi/uniorg][uniorg]] for the TS parsing path (Phase 1). Full org-element.el-equivalent parser in TypeScript, unified-ecosystem compatible, unist ASTs. Encapsulated behind a domain facade.

** org-ql Dependency Tree
org-ql needs ~10 elisp packages. Vendor only the minimal set for =org-ql-select=: =org-ql.el=, =peg.el=, =s.el=, =dash.el=, =f.el=, =ts.el=, =map.el=, =compat.el=. Strip all interactive UI code.

*** Design: The =org-parse= Facade

Two layers exist in =packages/org/= for reading org files, with a clean boundary:

- =org-reader.ts=: CRUD-oriented. Line-by-line state machine for reading/writing headings, PROPERTIES, body. Used by the org tool for mutations. Stays.
- =org-parse.ts= (NEW): Rich read-only parsing via uniorg. Full inline markup, links, tables, timestamps. Used for rendering, search, keyword extraction. Encapsulates uniorg entirely — no uniorg types leak into the public API.

These are permanently complementary. =org-reader= owns mutable file format. =org-parse= owns rich content understanding. Never plan to merge them — uniorg's roundtrip (parse→stringify) doesn't preserve formatting the way the line-by-line parser does.

#+begin_src typescript
/** Convert org-mode text to CommonMark markdown. */
export function orgToMarkdown(org: string): string;

/** Convert org-mode text to plain text (no markup). For token counting, search. */
export function orgToPlainText(org: string): string;

/** Extract #+KEYWORD frontmatter from org text. Returns Record<lowercase-key, value>. */
export function extractOrgKeywords(org: string): Record<string, string>;

/** Parse org headings with full structure. */
export interface OrgHeading {
  level: number;
  title: string;
  tags: string[];
  properties: Record<string, string>;
  body: string;
  children: OrgHeading[];
}
export function parseOrgHeadings(org: string): OrgHeading[];
#+end_src

Implementation details (internal, not exported):
- =orgToMarkdown=: parse → orgToHast → toMdast → toMarkdown. Handle PROPERTIES drawers explicitly: either strip them or convert to fenced code blocks pre-conversion.
- =orgToPlainText=: parse → toString (from orgast-util-to-string). One-liner.
- =extractOrgKeywords=: parse → walk AST for =keyword= type nodes at OrgData level. These are the =#+KEY: value= lines. No need for =uniorg-extract-keywords= plugin.
- =parseOrgHeadings=: parse → walk AST for =headline= type nodes, extract title (from title children via toString), tags, properties (from property-drawer children), body.

Re-export from =packages/org/src/index.ts=.

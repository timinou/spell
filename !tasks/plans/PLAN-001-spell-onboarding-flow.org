* Problem

When the QML tool is used in remote mode for the first time, there is no guided setup.
The user must manually build the Android app, install it, run ADB commands, configure the
server, and hope it all connects. The USAGE.md documents 6+ manual steps.

* Goal

First-time use of the QML tool with a connected Android device triggers a blocking TUI
modal ("Spell Setup") that:

1. Detects ADB and a connected device
2. Installs the pre-built Spell APK onto the phone (auto-built locally or downloaded from GitHub Releases)
3. Sets up ADB reverse port-forwarding
4. Launches the app on the phone
5. Waits for the WebSocket connection from Spell
6. Dismisses the modal and lets the agent proceed

The user's experience: plug in phone, see a setup modal, wait ~15 seconds, done.

* Rename

The Android app is renamed from "omp-remote" to "Spell" everywhere:
- Package: io.ohmypi.spell
- App name: Spell
- Directory: apps/spell/ (rename apps/omp-remote/)
- Binary/APK: spell.apk

* Architecture

** Flow diagram

#+begin_example
QmlTool.execute("launch", ...)
  |
  +-- session.qmlRemoteServer exists? --> yes --> use it, done
  |
  +-- no --> #ensureRemoteConnection(context)
              |
              +-- Is ADB available? (Bun.which("adb"))
              |     no --> throw ToolError with install instructions
              |
              +-- Is a device connected? (adb devices)
              |     no --> show modal: "Connect your Android phone via USB"
              |            poll until device appears (2s interval, 120s timeout)
              |
              +-- Is Spell installed? (adb shell pm list packages io.ohmypi.spell)
              |     no --> locate or acquire APK:
              |            1. Check local build: apps/spell/build/.../spell.apk
              |            2. Check toolsDir: ~/.omp/tools/spell.apk
              |            3. Try auto-build (if Qt Android toolchain detected)
              |            4. Download from GitHub Releases (like tools-manager)
              |            Then: adb install spell.apk
              |
              +-- Set up port forwarding: adb reverse tcp:9473 tcp:9473
              |
              +-- Start QmlRemoteServer on port 9473
              |
              +-- Launch Spell: adb shell am start io.ohmypi.spell/...
              |
              +-- Wait for WebSocket connection (30s timeout)
              |     modal: "Waiting for Spell to connect..."
              |     connection arrives --> modal: "Connected!" (auto-dismiss after 1s)
              |
              +-- Attach server to session.qmlRemoteServer
              |
              +-- Return, proceed with original tool call
#+end_example

** Blocking modal

Uses the existing ~context.ui.custom()~ factory (same mechanism as the login dialog and
ask tool). The factory returns a Container with DynamicBorder top/bottom, title, and a
dynamic content area that updates through each phase:

1. "Checking ADB..." -> "Device found: Pixel 7"
2. "Installing Spell..." (or "Spell already installed")
3. "Setting up connection..."
4. "Waiting for Spell to connect..."  (with a Loader spinner)
5. "Connected!" (green, auto-dismiss)

The user can press Escape at any point to cancel (returns ToolError to agent).

** APK acquisition (SpellManager)

New file: ~packages/coding-agent/src/tools/spell/manager.ts~

Follows the tools-manager.ts pattern:
- Checks ~/.omp/tools/spell.apk for a cached copy
- If not found, checks for a local build at apps/spell/build output
- If not found, attempts auto-build if Qt Android toolchain is detected (cmake + Qt6 + NDK)
- If not found, downloads from GitHub Releases (timinou/kika releases, asset "spell-arm64-v8a.apk")
- Stores downloaded/built APK in ~/.omp/tools/spell.apk

** ADB utilities

New file: ~packages/coding-agent/src/tools/spell/adb.ts~

Pure functions wrapping ADB commands via Bun shell ($``):
- ~isAdbAvailable()~: Bun.which("adb") !== null
- ~getConnectedDevices()~: parse `adb devices` output
- ~isSpellInstalled(deviceId?)~: `adb shell pm list packages io.ohmypi.spell`
- ~installApk(apkPath, deviceId?)~: `adb install <path>`
- ~setupPortForward(port, deviceId?)~: `adb reverse tcp:PORT tcp:PORT`
- ~launchSpell(deviceId?)~: `adb shell am start -n io.ohmypi.spell/.MainActivity`
- ~uninstallSpell(deviceId?)~: for updates

* Key Decisions

1. *Blocking modal via context.ui.custom()* — same proven pattern as LoginDialogComponent and ask tool. Agent execution pauses until connected or cancelled.
2. *APK in ~/.omp/tools/* — follows tools-manager convention for cached binaries.
3. *Auto-build + download fallback* — source builds preferred when toolchain available; GitHub Releases as fallback for packaged installs.
4. *Idempotent connection* — ensureSpellConnection is safe to call on every QML tool invocation; returns immediately if already connected.
5. *Escape to cancel* — every phase is cancellable; returns ToolError so the agent can report the failure.

* Problem

When spell sessions need input, approval, or have completed, the only signal is the niri overview overlay (which requires opening the overview). The workspace switcher in the DankBar shows no indication of which workspaces have sessions needing attention.

* Architecture

Two-part solution: spell writes status files, Quickshell reads and renders them.

** Communication protocol

Spell writes JSON status files to =~/.spell/status/<session-id>.json= on every status change. Format:

#+begin_src json
{
  "status": "running",
  "title": "Auth Refactor",
  "pid": 12345,
  "updatedAt": 1710000000000
}
#+end_src

Status values: ="idle"=, ="running"=, ="needs_input"=, ="error"=, ="completed"=, ="pending_approval"=.

=title= is the terminal window title set via OSC 0 (e.g. "Auth Refactor"). Quickshell matches this against niri window titles to determine which workspace each session is on.

The file is deleted on controller destroy (session exit). A stale-file reaper (>60s since updatedAt with no matching PID alive) handles crashes.

** Why title-based matching

All Ghostty windows share a single PID (4644 in the compositor's view), so PID-based workspace association doesn't work. Window titles are unique per tab/session and already set by spell. Quickshell's NiriService tracks =title= and =workspace_id= for every window.

* Key decisions

1. *File-based IPC* over Unix socket: simpler, Quickshell has =FileView=/=Process= but no easy Unix socket client for custom protocols. Polling at 2s is acceptable for a status indicator.
2. *Title matching* over PID matching: Ghostty multiplexes windows under one PID. Window titles are unique and already set by spell.
3. *Dots not badges*: Tiny colored circles are unobtrusive. No text labels — color alone communicates state. This matches the visual language of workspace "urgent" indicators.
4. *Only actionable states*: Running/idle don't get dots. The point is to surface states that need the user's attention.

* Problem

When spell sessions need input, approval, or have completed, the user must open the niri overview overlay to see status. The workspace switcher in the DankBar (Quickshell) shows no indication of which workspaces have sessions needing attention.

* Architecture

Two-part solution: spell writes status files per session, Quickshell reads them and renders colored dots on workspace buttons.

** Communication protocol

Spell writes JSON status files to =~/.spell/status/<niri-window-id>.json= on every status change:

#+begin_src json
{
  "status": "pending_approval",
  "windowId": 73,
  "pid": 9247,
  "updatedAt": 1710000000000
}
#+end_src

Status values: =idle=, =running=, =needs_input=, =error=, =completed=, =pending_approval=.

Keyed by niri window ID because Ghostty multiplexes all tabs under a single PID (verified: all 25 windows report =pid=4644=), so PIDs cannot distinguish windows. Niri window IDs are unique per window/tab.

** How spell discovers its niri window ID

At =NiriOverviewController= construction time, run =niri msg -j focused-window= via =Bun.$=. The controller is instantiated during =interactive-mode.init()= (line ~389 of =interactive-mode.ts=), at which point the terminal that launched spell IS the focused window. This returns ={ "id": N, "workspace_id": M, ... }=. Store the =id= as =#niriWindowId=.

This is a one-shot query at init, no ongoing tracking needed. The window ID is stable for the lifetime of the terminal tab.

** How Quickshell matches status files to workspaces

Quickshell's =NiriService.windows= array has objects with =.id= and =.workspace_id= for every niri window. The status file is keyed by =windowId=. Cross-reference: for each status file, look up the window in =NiriService.windows= by =id=, read its =workspace_id=.

** Why not PID-based matching

Ghostty is a single-process multi-window app. All windows report PID 4644 in niri's window data. The fish shell children under Ghostty have distinct PIDs, but niri only exposes the top-level app PID. Process tree walking is fragile and compositor-dependent.

** Why not title-based matching

Titles change (session title generated after first message), can be manually overridden by extensions (=setTitle=), and parsing is brittle. Window IDs are stable integers assigned by the compositor.

** ITEM Create SpellStatusService for Quickshell
:PROPERTIES:
:EFFORT: 1h
:END:

File: =~/.config/quickshell/dms/Services/SpellStatusService.qml=

A singleton service that:
1. Uses a =Process= to periodically (2s =Timer=) run a small script that reads all files in =~/.spell/status/=, parses each, and outputs a combined JSON object. Alternative: use =FileView= on a single manifest file (but polling a dir is simpler for multiple sessions).
2. Exposes =property var statusByWindowId: ({})= — maps niri window ID (number) to status string.
3. Provides a JS helper =function getWorkspaceStatuses(workspaceIdx)= that:
   - Iterates =NiriService.windows=

   - Returns array of status strings (only actionable ones: =needs_input=, =pending_approval=, =completed=, =error=)
4. Reaps stale files: if =updatedAt= is >120s old AND process with that PID is not alive, delete the file.

Process command (runs every 2s):
#+begin_src bash
find ~/.spell/status -name '*.json' -exec cat {} + 2>/dev/null | jq -s '.'
#+end_src
Or simpler: a Python/bash one-liner that outputs a JSON object.

Register as singleton: add =SpellStatusService { id: spellStatus }= in =DMSShell.qml= alongside other services.

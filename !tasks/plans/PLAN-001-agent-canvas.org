* Initial Message

Look, I just want you to be able to design your own interface when we discuss. Form of being to form of being, how do we make that happen?

* Problem

The agent has full QML infrastructure (bridge, armed tools, event loops, daemon mode) but never uses it during normal conversations. Visual communication — tables, diffs, architecture sketches, decision matrices — stays trapped in text. The agent can't design its own interface on the fly, despite having the rendering engine.

The gap isn't capability. It's vocabulary: no reusable components, no standard protocol, no skill teaching the agent when to reach for visual UI.

* Design

Three layers, built bottom-up:

** Layer 1: Canvas Runtime

A persistent QML window the agent spawns and updates incrementally via =send_message=.

*** Message Protocol

Standard message shapes between agent and canvas:

| Action    | Direction    | Purpose                                  |
|-----------+--------------+------------------------------------------|
| =set=     | Agent → QML  | Replace all content with new blocks      |
| =append=  | Agent → QML  | Add content blocks to existing canvas    |
| =remove=  | Agent → QML  | Remove a content block by id             |
| =prompt=  | Agent → QML  | Present a choice/input to the user       |
| =respond= | QML → Agent  | User's structured response to a prompt   |
| =sync=    | Agent → QML  | Request current canvas state             |
| =state=   | QML → Agent  | Canvas dumps its full content model      |

Content blocks are typed: ={ id, type, data }= where type maps to a component (=table=, =diff=, =tree=, =markdown=, =image=, =choice=).

*** Key Files
- =packages/coding-agent/src/modes/qml/canvas/Canvas.qml= — persistent canvas window, content block renderer, message dispatcher
- =packages/coding-agent/src/modes/qml/canvas/ContentBlock.qml= — polymorphic block that delegates to typed components

** Layer 4: Structured Input Protocol

*** ITEM Choice/input prompts via canvas
:PROPERTIES:
:EFFORT: 3h
:END:

The canvas =prompt= message presents choices to the user:

#+begin_src json
{
  "action": "prompt",
  "promptId": "arch-choice-1",
  "type": "radio",
  "question": "Which architecture?",
  "options": [
    {"value": "monolith", "label": "Monolith", "description": "..."},
    {"value": "microservices", "label": "Microservices", "description": "..."}
  ]
}
#+end_src

User response comes back as:
#+begin_src json
{"action": "respond", "promptId": "arch-choice-1", "value": "monolith"}
#+end_src

Supported prompt types: =radio= (single select), =checkbox= (multi), =rank= (drag to reorder), =text= (free input).

Must expose these queries:
- =blockCount= — number of content blocks
- =blocks= — full content model as JSON array (id, type, data per block)
- =blockById(id)= — single block by id
- =promptCount= — number of active prompts
- =promptById(promptId)= — prompt state by id

Must handle =reset= by clearing all blocks and prompts.

** ITEM Integration tests: canvas runtime protocol
:PROPERTIES:
:EFFORT: 2h
:END:

File: =packages/coding-agent/test/modes/qml-canvas-runtime.test.ts=
Harness: =packages/coding-agent/src/modes/qml/canvas/CanvasTestHarness.qml=
Skip: =describe.skipIf(!isBridgeAvailable())=

*** Protocol contracts

- Canvas starts with empty content model (=blockCount === 0=)
- =set= replaces entire content model — old blocks gone, new blocks visible via =blocks= query
- =append= adds blocks without disturbing existing ones (query =blockCount= before and after)
- =remove= by blockId deletes exactly that block, others survive
- =set= with empty content array clears all blocks

*** Ordering and idempotency

- Multiple rapid =set= calls — last one wins, no stale renders (send 3 sets within 10ms, query, verify last state)
- =set= with identical data is idempotent (no visual glitch, blockCount unchanged)
- =append= after =set= appends to the new content, not the old

*** Error resilience

- =remove= with nonexistent blockId is a no-op (no error, blockCount unchanged)
- =set= with unknown component type renders a fallback/error block (not crash). Query block, verify it exists with type indicator.
- Malformed payload (missing =action= field) does not crash canvas — stays in prior state
- Payload with action but wrong data shape (e.g. =set= without =content=) degrades gracefully

*** State sync

- =sync= action → canvas emits =state= event with full content model via bridge
- After =set= with 3 blocks, =get_state= returns exactly those 3 blocks with their types, ids, and data
- =get_state= after =remove= reflects the removal

** ITEM Integration tests: structured input protocol
:PROPERTIES:
:EFFORT: 1h
:END:

File: =packages/coding-agent/test/modes/qml-canvas-structured-input.test.ts=

*** Prompt/respond contracts

- Agent sends =prompt= with promptId and options → canvas renders selection widget (=findItems= with objectName =promptWidget=)
- User selects option → bridge event carries ={ action: 'respond', promptId, value }= (use =evaluate()= to select, =waitForBridgeEvent= to capture)
- Multiple concurrent prompts (different promptIds) track independently (send 2 prompts, respond to second, first still visible)
- Responding to a prompt disables the input widget (=findItems= after response, check =enabled= property)

*** Edge cases

- Prompt with single option still renders and is selectable (confirmation dialog)
- Prompt with no options renders a free-text input
- Second prompt with same promptId replaces the first (not duplicate)

*** Prompt types

- =radio= — single selection from options
- =checkbox= — multiple selections (value is array)
- =text= — free-text input, response carries typed text

*** Armed tool round-trip

- QML emits ={ _tool: 'write', _rid: 'r-1', path: '/tmp/test', content: 'x' }= → response arrives back in QML via =bridge.messageReceived= with matching =_rid=

Use =harness.waitForBridgeEvent()= to capture the =_tool= event, then =harness.simulateToolResponse()= with the =_rid=, then =harness.evaluate()= to verify the QML received the response.

* Risks and Mitigations

| Risk                                  | Mitigation                                           |
|---------------------------------------+------------------------------------------------------|
| Agent doesn't naturally use canvas    | Activation rule in system prompt + skill examples     |
| Component kit grows unbounded         | Start with 3, expand only after daily use validates   |
| Event storms from rapid interaction   | Debounce + armed tools for sync operations            |
| Session compaction loses canvas state | Accept for v1; sync protocol recovers from QML side   |
| No display available                  | Display detection + markdown fallback in skill        |
| Click simulation coupling             | Use evaluate() for v1; bridge mouse events deferred   |
| JSON.stringify key-order in dedup     | Document as known limitation; deep-equal adds weight  |

* Not In Scope (v1)

- Persistent canvas across session restarts (daemon handles window, conversation history handles state)
- Image generation in canvas (use existing =generate_image= tool, display as image block)
- Custom user-defined components (skill-provided components cover common cases)
- Collaborative canvas (multi-user) — single-agent, single-user only
- Bridge-level mouse event simulation for tests (use evaluate() workaround)
